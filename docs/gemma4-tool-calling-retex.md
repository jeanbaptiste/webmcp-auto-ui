# Gemma 4 — Tool Calling: Full Experience Report

> Document compiled from 5 sources: official Google docs, vLLM recipes, GitHub issue opencode #20995, Medium Google Cloud article, and the X/Twitter community. Date: 2026-04-13.

---

## Executive Summary

Gemma 4 uses a **proprietary tool calling protocol** based on special tokens (`<|tool_call>`, `<tool_call|>`, etc.) that is radically different from the OpenAI format. When going through an OpenAI compatibility layer (Ollama, vLLM in `/v1/chat/completions` mode), the translation is imperfect and a source of many bugs — particularly in streaming mode.

Key points to remember:
- The native format and the OpenAI-compatible format are **two distinct things** — do not confuse them
- vLLM requires `--tool-call-parser gemma4` and a dedicated Jinja template to work correctly
- Ollama <0.20.2 had a critical bug: tool_calls landed in the `reasoning` field instead of `tool_calls`
- In streaming mode, several AI SDK clients do not recognize Gemma 4 tool calls (bug opencode #20995)
- `enable_thinking=True` significantly improves tool calling accuracy
- Performance: 86.4% on tau2-bench (vs 6.6% for Gemma 3)

## 1. Native Gemma 4 Format (source: docs.google.dev)

### Special Tokens

Gemma 4 defines a tool calling protocol with dedicated tokens:

| Token | Role |
|-------|------|
| `<\|tool_call>` | Start of a function call |
| `<tool_call\|>` | End of a function call |
| `<\|tool_response>` | Start of a tool response |
| `<tool_response\|>` | End of a tool response |
| `<\|channel>thought\n...<channel\|>` | Reasoning block (thinking mode) |

### Message Structure (native format)

```
<|turn>system
[system prompt]<|turn>
<|turn>user
[user message]<|turn>
<|turn>model
[model response]<|turn|>
```

### Tool Declaration (native format)

```
<|tool>declaration:function_name{
  description:"Gets current temperature for a city",
  parameters:{
    properties:{
      city:{type:"STRING", description:"City name"},
      unit:{type:"STRING", enum:["celsius","fahrenheit"]}
    },
    required:["city"]
  }
}<tool|>
```

### Function Call Generated by the Model (native format)

```
<|tool_call>call:get_current_temperature{city:"Tokyo",unit:"celsius"}<tool_call|>
```

String-type parameters use pipe delimiters: `param:<|"|>value<|"|>`

### Tool Response Injected into Context

```
<|tool_response>response:get_current_temperature{temperature:22,condition:"Partly cloudy"}<tool_response|>
```

### Parsing Tool Calls (regex)

```python
import re
pattern = r'<\|tool_call>call:(\w+)\{(.*?)\}<tool_call\|>'
matches = re.findall(pattern, model_output)
```

### Declaration via JSON Schema (manual method)

```json
{
  "type": "function",
  "function": {
    "name": "get_current_temperature",
    "description": "Gets current temperature for a location",
    "parameters": {
      "type": "object",
      "properties": {
        "location": {
          "type": "string",
          "description": "City name, e.g. 'Tokyo'"
        },
        "unit": {
          "type": "string",
          "enum": ["celsius", "fahrenheit"]
        }
      },
      "required": ["location"]
    }
  }
}
```

### Declaration via Python Functions (automatic method)

```python
from transformers import AutoProcessor

# get_json_schema() automatically converts type hints
# and Google-style docstrings to JSON Schema
processor = AutoProcessor.from_pretrained("google/gemma-4-31B-it")
schema = processor.get_json_schema(my_function)
```

**Limitation**: complex nested objects (custom Config classes) lose detail during automatic conversion. Prefer manual JSON Schema in such cases.

## 2. OpenAI-Compatible Format via vLLM (source: docs.vllm.ai/projects/recipes)

### Supported Models

| Model | Params | Minimum GPU (BF16) |
|--------|--------|-------------------|
| Gemma 4 E2B IT | ~2B active | 1× 24 GB |
| Gemma 4 E4B IT | ~4B active | 1× 24 GB |
| Gemma 4 31B IT | 31B | 1× 80 GB |
| Gemma 4 26B-A4B IT (MoE) | 26B total / 4B active | 1× 80 GB |

### Launching the vLLM Server with Tool Calling

```bash
vllm serve google/gemma-4-31B-it \
  --max-model-len 8192 \
  --enable-auto-tool-choice \
  --tool-call-parser gemma4 \
  --reasoning-parser gemma4 \
  --chat-template examples/tool_chat_template_gemma4.jinja
```

**Critical flags:**
- `--enable-auto-tool-choice`: enables automatic tool call detection
- `--tool-call-parser gemma4`: dedicated parser that translates the native Gemma 4 format to the OpenAI format
- `--reasoning-parser gemma4`: for thinking mode support in parallel
- `--chat-template`: the official Jinja template is **required**; it is included in the official Docker container or available in the vLLM repo

### Tool Calling via OpenAI SDK (full flow)

```python
from openai import OpenAI
import json

client = OpenAI(base_url="http://localhost:8000/v1", api_key="EMPTY")

tools = [
  {
    "type": "function",
    "function": {
      "name": "get_weather",
      "description": "Get the current weather for a location",
      "parameters": {
        "type": "object",
        "properties": {
          "location": {
            "type": "string",
            "description": "City name, e.g. 'San Francisco, CA'"
          },
          "unit": {
            "type": "string",
            "enum": ["celsius", "fahrenheit"]
          }
        },
        "required": ["location"]
      }
    }
  }
]

# Step 1: Send the question with tools
response = client.chat.completions.create(
  model="google/gemma-4-31B-it",
  messages=[{"role": "user", "content": "What is the weather in Tokyo today?"}],
  tools=tools,
  max_tokens=1024
)

message = response.choices[0].message

# Step 2: Process the tool calls
if message.tool_calls:
  tool_call = message.tool_calls[0]
  print(f"Tool: {tool_call.function.name}")
  print(f"Args: {tool_call.function.arguments}")

  # Step 3: Send back the tool result
  response = client.chat.completions.create(
    model="google/gemma-4-31B-it",
    messages=[
      {"role": "user", "content": "What is the weather in Tokyo today?"},
      message,  # assistant message with the tool_call
      {
        "role": "tool",
        "tool_call_id": tool_call.id,
        "content": json.dumps({"temperature": 22, "condition": "Partly cloudy", "unit": "celsius"})
      }
    ],
    tools=tools,
    max_tokens=1024
  )
  print(f"Final answer: {response.choices[0].message.content}")
```

## 3. Differences: Native Format vs OpenAI-Compatible Format

| Aspect | Native Gemma 4 Format | OpenAI-Compatible Format |
|--------|---------------------|------------------------|
| Tool declaration | Tokens `<\|tool>declaration:...` | JSON in the `tools` field of the request |
| Function call | `<\|tool_call>call:name{args}<tool_call\|>` | `tool_calls` field in `message` |
| Tool response | `<\|tool_response>response:name{...}<tool_response\|>` | Message with `role: "tool"` and `tool_call_id` |
| Thinking | `<\|channel>thought\n...<channel\|>` | `reasoning_content` field (via vLLM) |
| Parsing | Custom regex | vLLM `gemma4` parser (automatic) |
| Abstraction level | Low-level, full control | High-level, compatible with OpenAI clients |

**Key point**: vLLM acts as a translator between the two formats. The `--tool-call-parser gemma4` and the Jinja template handle this translation. Without these two elements, vLLM will not recognize tool calls.

## 4. Known Issues

### Critical bug: streaming Ollama < 0.20.2 (Apple Silicon)

**Symptom**: tool_calls land in the `reasoning` field instead of the `tool_calls` field.

**Cause**: bug in Ollama's streaming that routes the response to the wrong field.

**Fix**: upgrade to Ollama ≥ 0.20.2. PR ollama/ollama#15306 fixed this issue.

**Reference**: Medium article (llama.cpp workaround), GitHub issue #20995

---

### Bug: Flash Attention freeze on Apple Silicon (Ollama)

**Symptom**: generation hangs indefinitely on prompts > 500 tokens with Flash Attention enabled.

**Cause**: Flash Attention incompatibility with long prompts on Apple Silicon (Ollama 0.20.3).

**Fix**: use llama.cpp directly with `-np 1` and without Flash Attention, or upgrade Ollama.

**Reference**: Medium article

---

### Bug: tool calls not recognized in streaming (opencode #20995)

**Symptom**: the model via Ollama correctly returns `tool_calls` (confirmed via direct curl), but the AI SDK (`@ai-sdk/openai-compatible`) does not recognize them in streaming mode. The model responds "I do not have the capability to execute system commands" instead of calling the tool.

**Cause**: the AI SDK compat layer `toolParser` does not correctly handle Gemma 4 tool calls in streaming — format differences in the deltas.

**Workaround**: use a different model (e.g. qwen3.5:9b) or wait for PR #16531 to be merged.

**Issue diagnosis**: the Ollama API works correctly (non-streaming AND streaming) — the problem is on the client SDK side.

```bash
# Direct verification (works correctly)
curl -s http://localhost:11434/v1/chat/completions -d '{
  "model": "gemma4:e4b",
  "messages": [{"role": "user", "content": "list files in /tmp"}],
  "stream": false,
  "tools": [{"type": "function", "function": {"name": "bash", "description": "Execute a shell command", "parameters": {"type": "object", "required": ["command"], "properties": {"command": {"type": "string"}}}}}]
}'
# Returns: finish_reason: "tool_calls", tool_calls: [{function: {name: "bash", arguments: "{\"command\":\"ls /tmp\"}"}}]
```

**Reference**: GitHub anomalyco/opencode#20995 (opened April 4, 2026)

---

### vLLM 0.19.0 incompatibility with NVIDIA Blackwell (GB10)

**Symptom**: ImportError on vLLM startup on an NVIDIA Blackwell GPU.

**Cause**: PyTorch 2.10.0 bundled in vLLM 0.19.0 is incompatible with PyTorch 2.11.0+ (CUDA Blackwell), different ABI.

**Fix**: use vLLM nightly wheels (`--extra-index-url https://wheels.vllm.ai/nightly/cu129`) or the official Docker image `vllm/vllm-openai:gemma4`.

**Reference**: Medium article

---

### llama.cpp pitfall: `-hf` flag downloads the vision projector (1.1 GB)

**Symptom**: OOM crash on startup with Gemma 4 26B MoE running locally.

**Cause**: the `-hf` (HuggingFace auto-download) flag silently downloads the 1.1 GB vision projector on top of the model.

**Fix**: use the direct path `--model /path/to/model.gguf` instead of `-hf`.

**Reference**: Medium article

---

### Session timeout during long tool calling cycles

**Symptom**: the session closes during a multi-step tool calling cycle.

**Cause**: Gemma 4 tool calling cycles can take 99 seconds on a Mac M4 Pro. The default Codex CLI timeout kills the session.

**Fix**: configure `stream_idle_timeout_ms: 1800000` (30 minutes).

**Reference**: Medium article

---

### Non-function tool types rejected by some servers

**Symptom**: error if `web_search_preview` (non-standard type) is in the tools list.

**Cause**: llama.cpp and some inference servers reject non-function tool types.

**Fix**: disable non-standard tool types (e.g. `web_search = "disabled"` in Codex CLI), or use `wire_api = "responses"`.

**Reference**: Medium article

## 5. Best Practices

### General

1. **Always validate** function names and arguments before execution — the model can hallucinate non-existent function names
2. **Do not use `globals()`** to resolve functions in production; prefer an explicit mapping `{"function_name": actual_function}`
3. **Detailed descriptions** for each parameter — Gemma 4 relies heavily on these to understand what it should pass
4. **Manual JSON Schema** for complex objects — automatic conversion from Python type hints loses details of nested classes
5. **`enable_thinking=True`** significantly improves tool calling accuracy (the model reasons before calling)

### For vLLM in Production

```bash
# Complete recommended configuration
vllm serve google/gemma-4-31B-it \
  --max-model-len 32768 \
  --gpu-memory-utilization 0.90 \
  --enable-auto-tool-choice \
  --tool-call-parser gemma4 \
  --reasoning-parser gemma4 \
  --chat-template examples/tool_chat_template_gemma4.jinja \
  --tensor-parallel-size 2  # for 31B on 2× A100
```

### For Local llama.cpp (Apple Silicon)

```bash
# Critical flags
./llama-server \
  -m /path/to/gemma-4-26b-a4b-it-q4_k_m.gguf \
  --jinja \       # enables the tool calling template
  -np 1 \         # single slot for KV cache (required for tool calling)
  -ctk q8_0 \     # KV cache quantization (940MB -> 499MB)
  -ctv q8_0 \
  -c 32768        # minimum recommended context
# DO NOT use -hf (downloads vision projector)
```

### Model Selection

| Situation | Recommendation |
|-----------|---------------|
| Maximum accuracy | 31B Dense or 26B MoE with thinking enabled |
| Speed (Apple Silicon) | 26B MoE Q4_K_M (activates only 3.8B params/token) |
| Limited GPU (24 GB) | E4B-it (effectively ~4B active params) |
| Server production | vLLM with 31B or 26B MoE on 80 GB GPU |

## 6. Recommendations for Our WebMCP Stack

### Context

Our stack uses `GemmaProvider` from the `@webmcp-auto-ui/agent` package, which relies on LiteRT (in-browser WASM) on the main thread. The targeted Gemma 4 models are the E2B/E4B variants (the only ones small enough to run in-browser).

### What Our Docs Do vs What Sources Recommend

| Aspect | Our current stack | What sources recommend |
|--------|---------------------|--------------------------------|
| Tool format | Probably standard OpenAI via `GemmaProvider` | Use the official Jinja template + `gemma4` parser if via vLLM |
| Thinking mode | Not documented for Gemma 4 | `enable_thinking: true` in `chat_template_kwargs` — improves accuracy |
| Streaming tool calls | Via our agent loop | Risk of non-detection (bug opencode #20995) — validate that our parser handles Gemma 4 deltas |
| Argument validation | Not documented | Add validation before execution |
| Tool descriptions | Standard MCP format | Ensure descriptions are sufficiently detailed (Gemma 4 depends on them) |
| Timeout | Not documented | Configure long timeouts for multi-tool cycles (100s+ observed) |

### Recommended Actions

1. **Verify our streaming handling**: bug opencode #20995 shows that the AI SDK does not always detect Gemma 4 tool_calls in streaming. Audit our `runAgentLoop` to ensure it correctly parses Gemma 4 streaming deltas.

2. **Test `enable_thinking`** for Gemma 4 models via our `RemoteLLMProvider` (vLLM server) — potentially much better accuracy score.

3. **If we deploy Gemma 4 via vLLM**: do not forget the 3 required flags (`--enable-auto-tool-choice`, `--tool-call-parser gemma4`, `--chat-template`) + use the official Docker image `vllm/vllm-openai:gemma4`.

4. **Ollama versions**: if we support Ollama as a backend, ensure the user has ≥ 0.20.2. Below that, tool_calls arrive in the wrong field.

5. **MoE vs Dense**: for local use on Apple Silicon, the 26B MoE is faster than the 31B Dense because only ~4B params are active per token. Document this choice in our `GemmaProvider`.
